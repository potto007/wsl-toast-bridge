wsl-toast-bridge.

Native Windows toast notifications from WSL — and clicks that come back: focus the terminal, focus the exact Herdr pane, never open a stray PowerShell window.

The problem

WSL has no notification daemon, so notify-send either doesn't exist or talks to nobody. Tools that emit desktop notifications — Herdr with delivery = "system", build scripts, monitors — fail silently. And the usual fix, BurntToast's New-BurntToastNotification, has a sting: clicking the toast activates the PowerShell AppId and opens a useless PowerShell window.

Pane resolution

Herdr's notify-send call carries only text — no pane id. The shim closes the gap at send time: it identifies Herdr by parent process (ps -o comm= -p $PPID), then matches the notification text against herdr agent list — each agent's terminal title and repo directory name. Exactly one hit → that pane_id rides along in the launch URI. Zero or many → the click still focuses the terminal, and Herdr's own prefix+o jumps to the notification target.

Three bugs you don't have to rediscover

1 · wsl.exe silently no-ops in a hidden window

From a windowless protocol handler, bare wsl.exe exits 0 having run nothing — there's no console to attach. focus.ps1 starts it via ProcessStartInfo with redirected stdout/stderr, which hands it usable stdio and makes it actually execute.

2 · Bad image paths warn and exit 0

BurntToast checks the file, prints WARNING: … failed back to icon, and succeeds anyway. A /mnt/c/… path is invisible to Windows, so your logo silently becomes burnt toast. toast converts through wslpath -w and warns on non-files; every stage logs (~/.local/state/*.log, %LOCALAPPDATA%\herdr-toast\focus.log).

3 · The versioned pwsh path breaks on update

(Get-Command pwsh).Source resolves to WindowsApps\Microsoft.PowerShell_7.6.4.0_… — dead after the next Store update. The protocol registration uses the stable %LOCALAPPDATA%\Microsoft\WindowsApps\pwsh.exe alias.

Why not New-BurntToastNotification?

It pins toast activation to the PowerShell AppId — the click is "launch PowerShell". The fix is one layer down the same module: New-BTContent -ActivationType Protocol -Launch <uri> routes the click to any registered URL protocol instead. Same toast, same images, controllable click.
